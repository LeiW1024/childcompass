01 — Idea Recap
The Problem We're Solving
😓
The Pain
Information is scattered & hard to find
Parents in Erfurt had no central platform to discover age-appropriate activities for their young children. Information was spread across dozens of websites, Facebook groups, and word of mouth.
👨‍👩‍👧
Who Is It For?
Parents with children ages 0–6 in Erfurt
Young families looking for daycare, playgroups, sports, arts, music, language classes, swimming, and nature activities — all within their neighbourhood.
📍
The Insight
Parents think geographically, not by category
"What can my 2-year-old do on Saturday morning near our home?" — proximity and age-fit matter more than browsing abstract categories.
🧭
The Solution
One platform — map, filter, book
ChildCompass puts every local activity on a live map. Filter by age, price, distance and category. Book a spot with one click. Providers manage listings and respond to requests.
02 — Critical Hypothesis & Outcome
What We Believed — and What We Learned
🧪 Most Critical Hypothesis
"If parents can see activities on a map filtered by their child's age, they will find relevant options faster than scattered Google searches — AND they are willing to book directly online instead of calling providers."
How We Tested It
🔬 Prototype + Real User Interviews
Built a working prototype and talked to many parents in Erfurt. Every parent shared the same frustrations — scattered information, no central platform, hard to find age-appropriate options nearby.
What Was the Result
✅ Both Hypotheses Validated
Map + age filter proved effective for discovery. Booking flow works end-to-end on Vercel. Biggest real-world challenge: accurate provider data — coordinates and activity details had to be manually researched and verified.
What Changed & What I Learned
💡 Data & Users First
Technology wasn't the hard part — data quality was. Wrong pins destroy trust faster than any UX problem. And talking to parents before writing code gave the confidence to keep building — they confirmed everything.
03 — The Journey
Three Decisions That Shaped the Product
🧪
Prototype Problem
"Dummy data didn't convince anyone — not even me"
The prototype proved the concept but the UI was rough and fake activities felt hollow. Real trust needs real data.
→ Decision: Build a real data pipeline before polishing the UI
🤖
The Pivot
"Manual research was too slow — so I built an AI scraper"
Finding real provider details (coordinates, prices, schedules) manually took hours per listing. The Claude-powered web scraper automated this — scaling from 17 to 134 listings.
→ Decision: Let AI solve the data problem, not manual effort
🚀
Validated on Vercel
"Testing on Vercel was not optional — it was essential"
Running locally is not enough. Only by deploying to Vercel could I verify that the full product — map, booking, auth, data — works in a real production environment for real users.
→ Decision: Always validate in production — not just on localhost

TDD — Red → Green → Refactor
Each agent writes tests first: DB → repositories.test.ts · Backend → api.test.ts · Frontend → component.test.tsx
⚡
Sequential or Parallel — depends on the feature
Full feature (DB→API→UI) runs sequentially. Independent tasks — e.g. "fix map zoom" + "add pagination" — run simultaneously across branches.
● What made the process more effective?
🎯
Focused context per agent
Each agent loads only its domain rules — no overload, no cross-layer confusion.
🚧
Hard boundaries enforced
DB never writes API routes. Frontend never touches Prisma. Bugs stayed isolated.
📋
CLAUDE.md + rules.md always loaded
9 rule files auto-load — consistent standards across all agents, every session.
🔁
TDD caught bugs at the boundary
Tests written before code — CI gate blocked broken PRs before merging.
07 — AI Features in the Product

Claude Code Web Search
claude-sonnet-4-5 · built-in web_search
Admin selects desired activity categories (e.g. DAYCARE, SPORTS). Claude uses the built-in web_search tool to find providers in Erfurt, extracts structured listing data (title, price, age range, address), and imports directly into the database.

n8n Workflow Chatbot
n8n · Google Gemini AI
A floating chat bubble available on every page. Parents type a question — the Next.js API forwards it to an n8n webhook, which passes it to Google Gemini with a ChildCompass system prompt, and streams the reply back into the chat widget.
